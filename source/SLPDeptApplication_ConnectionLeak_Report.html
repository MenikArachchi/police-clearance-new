1. Executive Summary

When SLPDeptApplication runs on the server, the number of active MySQL connections keeps growing until the database connection pool is exhausted and the application fails with a "cannot connect to database" error.

Code review identified three confirmed root causes:

  1. The DatabaseManager class initialises the datasource twice on every application startup due to a duplicate static block.
  2. Most critical: The DailyTransactionReportDAO opens a brand-new database connection for every single row returned by the Daily Transaction Report query. If the report returns 200 rows, 200 extra connections are opened in a tight loop.
  3. The CertificateIssuanceBusiness.updateApplicationCertificatePrintedStatus() method closes the pooled connection twice — once inside the try body and again in the finally block — which can corrupt the pool's internal connection tracking.

4. Fix 2 — New Connection Opened Per Result Row CRITICAL

SLPCommonFramework/src/main/java/lk/icta/police/framework/dao/DailyTransactionReportDAO.java

fetchApplicationVerificationList() receives a Connection conn from its caller. While iterating the report's ResultSet, it calls getConstructedApplicationFromResultSet() for each row, which internally calls getApplicationPreviousReferenceNoByReferenceNo(). That private method opens a completely new DB connection from the pool for every single row.

If the report returns 500 rows and 3 users run the report at the same time, that is 1,500 extra connections opened in seconds — exhausting any reasonable pool size.
